# 医药合规审查

对医药企业的文本内容进行合规性审查，识别潜在的职业风险、商业贿赂风险及学术推广合规性问题。

## 适用场景

当用户提供以下任何一种内容并要求合规审查时使用本技能：

- 岗位描述（JD）— 招聘网站或内部岗位说明书
- 制度文件 — 管理办法、操作规程、考核方案
- 推广材料 — 学术推广方案、会议邀请、宣传文案
- 销售相关文档 — 销售指标、绩效考核、激励方案
- 任何涉及医药代表、学术推广、处方药营销的文本

## 核心原则

1. **每条违规必须给出原文证据** — 直接引用原文片段，不做模糊指控
2. **无违规时明确标注合规** — 不过度解读，不制造恐慌
3. **区分严重性** — high（直接违法违规）、medium（存在风险隐患）、low（表述不够规范）
4. **给出可操作的修改建议** — 具体说明如何修改，而非笼统建议

## 合规逻辑模型

即使内容未命中黑名单关键词，若违反以下逻辑准则，亦判定为不合规：

### 身份偏移逻辑

医药代表职责仅限于学术传递、沟通及反馈。以下行为视为违规：

- 涉及药品销售、收款、处理购销票据
- 将医药代表定位为"销售"角色
- 要求完成销售指标、销售任务

### 处方干预逻辑

严禁以任何形式干预医生处方行为：

- 收集、核对、追踪特定医生的处方量
- 将推广行为与医生个人开方量挂钩
- 使用"统方"、"方量统计"、"处方量追踪"等工具
- 设定与处方量相关的考核指标
- 要求"提高产出"、"上量"、"带量"

### 利益交换逻辑

严禁通过利益手段换取销量或进院：

- 通过赞助、捐赠、课题合作预设与销量相关的条件
- "捐赠换销量"、"定向捐赠"
- 回扣、返点、提成、好处费
- 以"客情维护"、"关系维护"名义的利益输送
- 过票、虚开发票

### 非学术化推广逻辑

所有推广活动必须在学术场景进行：

- 严禁在非学术场景（娱乐场所、餐厅、旅游景点）开展活动
- 严禁"餐桌会"、"客情宴请"、"套会"
- 严禁通过"促进关系"掩盖利益输送

### 科学严谨性逻辑

学术内容必须基于循证医学：

- 严禁隐瞒已知的不良反应风险
- 严禁使用夸大、绝对化或误导性的疗效描述
- 禁用词：首选、最佳、根治、安全无毒、无效退款
- 严禁未经验证的疗效宣称

## 黑名单关键词

### 经济利益/贿赂
回扣、好处费、提成、返点、答谢费、送礼、辛苦费、茶水费、红包、返利、感谢费、定向捐赠、捆绑销售、过票、虚开发票、捐赠换销量、业务招待礼品、客情维护、关系维护

### 销量/统方统计
统方、TF、上量、销量、带量、处方量、冲刺、方量统计、处方量追踪、提高产出、抢收、纯销、公司获益、产品获益、冲量、走量

### 接待/娱乐风险
促进关系、餐桌会、旅游、度假、景点门票、招待、娱乐、客情宴请、站台、代签到、套会

### 绝对化/违法词汇
首选、最佳、根治、安全无毒、无效退款、最低价、最便宜、最流行、最受欢迎

## 工作流

### 1. 识别输入类型

确定用户提供的内容类型：

- 直接粘贴的文本
- 单个文件（docx/pdf/txt/pptx/ppt/xlsx/xls/csv/tsv）— 需要先提取文本内容
- 多个文件 — 逐个提取文本后合并审查
- 目录路径 — 递归扫描所有支持格式的文件

支持的文件格式：
- `.docx` / `.doc` — Word 文档，提取段落和表格文本
- `.pdf` — PDF 文档，提取可选择文本
- `.pptx` / `.ppt` — PowerPoint 演示文稿，提取每页幻灯片的标题、正文、备注和表格文本
- `.xlsx` / `.xlsm` / `.xltx` / `.xltm` / `.xls` — Excel 工作簿，按工作表提取每行单元格文本
- `.csv` / `.tsv` — 表格文本文件，按行提取单元格文本
- `.txt` / `.md` — 纯文本文件，直接读取

如果是文件，先提取全部文本内容（包括表格中的文本），然后按以下流程审查。

### 2. 内容预处理

- 清除 HTML 标签，保留纯文本
- 如果内容超过 2000 字，按段落分块处理
- 保留原始段落结构，不要合并或拆分语义完整的段落
- 记录每个内容块的来源（文件名、段落位置）
- **PPT 文件**：按幻灯片页码组织内容，每页作为一个独立段落块，格式为 `[第X页] 标题\n正文内容\n备注`
- **多文件处理**：每个文件独立提取和标记来源，审查结果中标注文件名，最终汇总为一份报告

### 3. 逐条审查

对每条内容执行以下检查：

1. **关键词扫描** — 检查是否命中黑名单关键词
2. **逻辑模型匹配** — 检查是否违反五大合规逻辑
3. **上下文分析** — 结合上下文判断是否构成实质违规（避免误报）
4. **严重性评估** — 根据违规性质判定 high/medium/low
5. **证据提取** — 直接引用原文中的违规片段
6. **建议生成** — 给出具体的修改方案

### 4. 输出审查结果

对每条内容输出结构化 JSON 结果，格式如下：

```json
{
  "results": [
    {
      "content_index": 1,
      "content_summary": "内容摘要（20字以内）",
      "is_compliant": true,
      "risk_level": "none|low|medium|high",
      "violations": [
        {
          "rule": "违反的规则名称（如：身份偏移逻辑/处方干预逻辑/利益交换逻辑/非学术化推广逻辑/科学严谨性逻辑/黑名单关键词）",
          "evidence": "原文中的违规证据（直接引用原文片段）",
          "description": "违规说明",
          "severity": "low|medium|high",
          "suggestion": "修改建议"
        }
      ]
    }
  ]
}
```

字段说明：
- `content_index`：内容序号（从1开始）
- `content_summary`：20字以内的内容摘要
- `is_compliant`：合规判定，true 为合规，false 为不合规
- `risk_level`：风险等级，取所有违规中最高 severity；无违规时为 "none"
- `violations`：违规列表，无违规时为空数组 `[]`
- `violations[].rule`：违反的具体规则名称
- `violations[].evidence`：**必须直接引用原文片段**作为证据
- `violations[].description`：违规行为的具体说明
- `violations[].severity`：该条违规的严重性（high/medium/low）
- `violations[].suggestion`：具体的修改建议

输出要求：
1. 每条违规必须在 evidence 字段中给出原文片段作为证据
2. 无违规时 violations 为空数组，is_compliant 为 true，risk_level 为 "none"
3. 有违规时 is_compliant 为 false，risk_level 取所有违规中最高 severity
4. 严格输出 JSON，不要输出任何其他解释文字

### 5. 生成汇总报告

如果审查多条内容，在最后生成汇总：

```
## 📊 审查汇总

- 审查总数：X 条
- ✅ 合规：X 条
- ❌ 不合规：X 条
- 🚨 违规条目：X 条

### 风险分布
- 高风险：X 条
- 中风险：X 条
- 低风险：X 条

### 高频违规规则
1. 身份偏移逻辑：X 次
2. 处方干预逻辑：X 次
...

### 整体建议
[基于审查结果给出整体性的合规改进建议]
```

## 判定边界

### 以下情况判定为合规

- 岗位职责描述为"学术推广"、"医学信息传递"、"学术交流"
- 考核指标为"学术覆盖率"、"拜访频次"、"学术活动场次"
- 使用"医学信息专员"、"医学联络官"等合规岗位名称
- 推广活动在学术场景进行（学术会议、院内讲座）

### 以下情况判定为不合规

- 岗位名称含"销售"但职责涉及处方药推广
- 考核指标含"销售额"、"销量"、"处方量"
- 职责描述含"开发客户"、"维护客情"、"完成指标"
- 推广方式含"餐桌会"、"客情宴请"

### 需要结合上下文判断的灰色地带

- "完成业绩目标" — 如果上下文明确是学术覆盖目标则合规
- "客户拜访" — 如果是学术拜访则合规，如果是销售拜访则不合规
- "市场开发" — 如果是学术市场教育则合规，如果是销售渠道开发则不合规

对灰色地带内容，标注为 medium 风险并说明判断依据。

## 文件处理指引

当用户提供文件时：

1. **docx/doc 文件** — 提取所有段落文本和表格内容
2. **pdf 文件** — 提取可选择文本；扫描件需 OCR
3. **pptx/ppt 文件** — 逐页提取：幻灯片标题、文本框内容、表格文本、备注栏内容。每页视为一个独立内容块
4. **xlsx/xls/csv/tsv 文件** — 按工作表或表格行提取单元格文本，保留工作表名和行号
5. **txt/md 文件** — 直接读取
6. **目录** — 递归扫描所有支持格式的文件

### 多文件处理策略

当用户一次提供多个文件时：

1. 逐个文件提取文本内容
2. 每个文件的内容独立标记来源（文件名）
3. 所有文件的内容合并后统一审查
4. 审查结果中每条违规标注所属文件
5. 最终输出一份汇总报告，按文件分组展示结果

报告格式示例：

```
## 📄 文件一：xxx.docx
[该文件的审查结果]

## 📄 文件二：xxx.pptx
[该文件的审查结果]

## 📊 汇总
[所有文件的整体统计]
```

### PPT 文件处理细节

PPT 文件的每一页幻灯片可能包含：
- 标题文本框
- 正文文本框（可能有多个）
- 表格
- 备注栏（演讲者备注）
- SmartArt 或图表中的文本

提取时将每页组织为：

```
[第1页] 幻灯片标题
正文内容（合并所有文本框）
表格内容（如有）
备注：演讲者备注内容（如有）
```

PPT 中的图片无法提取文本，如果关键信息在图片中，标注为"[图片内容无法提取，需人工审查]"。

提取后按上述工作流逐条审查，最终输出完整的审查报告。

## 质量标准

好的审查结果应该：

- 让合规团队能直接定位问题所在
- 让业务团队知道具体怎么改
- 不遗漏明显的合规风险
- 不对合规内容产生误报
- 证据充分、建议可操作

## 脚本工具

本技能附带 `scripts/` 目录，包含文件解析和报告生成脚本：

### extract_file.py

通用文件文本提取工具，支持 docx、pptx、pdf、txt、doc、xlsx、xls、csv、tsv 格式。

```bash
# 安装依赖
pip install -r scripts/requirements.txt

# 提取单个文件
python scripts/extract_file.py document.docx

# 提取多个文件
python scripts/extract_file.py file1.docx file2.pptx file3.pdf file4.xlsx

# 扫描目录
python scripts/extract_file.py --dir /path/to/folder
```

输出为 JSON 格式（stdout），每个文件一条记录：

```json
[
  {
    "title": "文件名（不含扩展名）",
    "source": "文件完整路径",
    "content": "提取的全部文本内容",
    "pages": [{"page": 1, "title": "...", "content": "..."}],
    "format": ".docx"
  }
]
```

Excel 文件的 `pages` 字段用于保存工作表内容，结构为：

```json
[
  {
    "sheet": 1,
    "title": "工作表名称",
    "content": "[工作表1] 工作表名称\n第1行: ..."
  }
]
```

### generate_report.py

将审查结果 JSON 转为可视化 HTML 报告。

```bash
# 从 JSON 文件生成 HTML
python scripts/generate_report.py results.json

# 指定输出路径
python scripts/generate_report.py results.json --output report.html

# 从 stdin 读取
echo '{"results":[...]}' | python scripts/generate_report.py --stdin --output report.html
```

输入 JSON 格式：

```json
{
  "results": [
    {
      "content_index": 1,
      "content_summary": "内容摘要（40字以内）",
      "source_file": "来源文件名（可选）",
      "is_compliant": false,
      "risk_level": "none|low|medium|high",
      "violations": [
        {
          "rule": "违反的规则名称（如：身份偏移逻辑/处方干预逻辑/利益交换逻辑/非学术化推广逻辑/科学严谨性逻辑/黑名单关键词）",
          "evidence": "原文中的违规证据（直接引用原文片段）",
          "description": "违规说明",
          "severity": "low|medium|high",
          "suggestion": "修改建议"
        }
      ]
    }
  ]
}
```

### 完整工作流示例

```bash
# 1. 提取文件内容
python scripts/extract_file.py --dir /path/to/jd > extracted.json

# 2. Agent 对 extracted.json 中的内容执行合规审查，输出 results.json

# 3. 生成 HTML 报告
python scripts/generate_report.py results.json --output compliance_report.html
```

Agent 可以调用这些脚本完成端到端的文件审查流程。
